# Source of Truth

> Figma is the source of truth for design decisions. Code is the source of truth for runtime behavior. The vault must reflect both.

## Statement

Three artefacts coexist; each is canonical for a different question.

| Artefact | Canonical for |
|---|---|
| Figma | What the system **should be** — design intent, visual decisions, component contracts at design time. |
| Code (`packages/*/src/**`) | What the system **is** — actual runtime behavior, real prop names, real values. |
| Vault | The LLM-facing description of both. Must reflect the code; cross-references Figma. |

## Rationale

A design system has two truthful layers that can disagree:

- **Designer intent** is captured in Figma. When a designer changes a Figma component, the system's intended shape has changed — even if code hasn't caught up yet.
- **What ships** is the code. Consumers import the package, the code runs, that's the system.

The vault has to pick one as primary for any given fact. The pragmatic rule: **vault tracks code, not Figma.** Figma drift becomes a tracked task (a "Figma → code sync" workflow), not an in-vault contradiction.

## Implications

- When the vault disagrees with code, the vault is wrong. Update the vault.
- When Figma disagrees with code, **either** can be wrong. The team decides whether to update Figma (code is right) or update code (Figma is right) — that decision becomes a task.
- The vault does not duplicate Figma's contents. It links out to Figma nodes via the `figma/` folder and `source:` frontmatter entries.

## Counter-examples

- ❌ A vault page that documents prop names from an old Figma version while the code uses new prop names.
- ❌ A vault page that invents prop names that exist in neither Figma nor code.
- ✅ A vault page whose props section was generated by reading `MDSButton.tsx`, with a Figma node link for the visual reference and a `last-synced:` date proving freshness.
